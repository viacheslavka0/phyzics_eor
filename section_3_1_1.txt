### 3.1.1. Архитектура электронного образовательного ресурса

Разрабатываемый электронный образовательный ресурс построен по классической клиент-серверной архитектуре. Серверная часть отвечает за хранение данных, проверку ответов учеников и управление учебными сессиями. Клиентская часть обеспечивает взаимодействие пользователя с системой через веб-интерфейс. Такое разделение позволяет независимо развивать функциональность каждой части и упрощает масштабирование системы при росте числа пользователей.

Серверная часть реализована на основе фреймворка Django версии 4.x. Выбор этого фреймворка обусловлен его развитой системой работы с базами данных через объектно-реляционное отображение (ORM), встроенной системой аутентификации и возможностью быстрого создания административного интерфейса. Для организации взаимодействия с клиентской частью используется подсистема Django REST Framework, которая позволяет описывать прикладной программный интерфейс (API) в виде набора представлений (ViewSet) и сериализаторов (Serializer). Все данные передаются между клиентом и сервером в формате JSON через стандартный протокол HTTP.

Структура базы данных отражает логику построения учебного курса и реализуемых этапов обучения. В основе лежит иерархия «класс – раздел – тема – система знаний». Каждая система знаний содержит собственные средства для осмысления (изображение таблицы, зоны, вопросы, тексты с пропусками), метод решения задач и набор связанных заданий. Для фиксации хода обучения введены модели учебной сессии, попыток осмысления и попыток решения задач. Они связываются с конкретным пользователем и конкретной системой знаний, что позволяет отслеживать прогресс ученика на каждом этапе.

Центральной моделью в структуре данных является модель «Система знаний» (KnowledgeSystem). Она связывает все элементы учебного процесса: изображения для осмысления, вопросы различных типов, тексты с пропусками, метод решения и задачи. В листинге 3.1 показан фрагмент описания этой модели, демонстрирующий основные поля и связи.

Листинг 3.1 – фрагмент модели системы знаний

```python
class KnowledgeSystem(models.Model):
    topic = models.ForeignKey(Topic, on_delete=models.CASCADE, 
                              related_name="knowledge_systems")
    title = models.CharField(max_length=255)
    description = models.TextField(blank=True)
    comprehension_image = models.ImageField(upload_to="ks/comprehension/",
                                           blank=True, null=True)
    comprehension_pass_threshold = models.PositiveIntegerField(default=85)
    typical_task_title = models.CharField(max_length=255, blank=True)
    typical_task_description = models.TextField(blank=True)
    status = models.CharField(max_length=10, choices=STATUS_CHOICES, default="draft")
```

Модель содержит ссылку на тему, к которой относится система знаний, описание для ученика, изображение таблицы для этапа осмысления и порог успешности прохождения этого этапа. Также в модели хранится формулировка типовой задачи, которая помогает ученику понять, для решения каких задач применяется данная система знаний.

Для взаимодействия клиентской и серверной частей разработаны REST API endpoints. Они организованы в виде набора маршрутов, которые регистрируются через механизм роутера Django REST Framework. Endpoints разделены на две группы: для ученика и для учителя. Ученику доступны endpoints для просмотра каталога систем знаний, получения данных о системе знаний, управления учебной сессией и решения задач. Учителю доступны endpoints для создания и редактирования структуры курса, загрузки изображений, создания вопросов и управления задачами. В листинге 3.2 показан фрагмент конфигурации маршрутов, демонстрирующий структуру API.

Листинг 3.2 – фрагмент конфигурации API маршрутов

```python
router = DefaultRouter()
router.register(r"catalog", CatalogViewSet, basename="catalog")
router.register(r"ks", KnowledgeSystemViewSet, basename="ks")
router.register(r"session", LearningSessionViewSet, basename="session")
router.register(r"task", TaskViewSet, basename="task")
router.register(r"teacher/sections", TeacherSectionViewSet, basename="teacher-sections")
router.register(r"teacher/topics", TeacherTopicViewSet, basename="teacher-topics")
router.register(r"teacher/ks-full", TeacherKnowledgeSystemFullViewSet, basename="teacher-ks-full")
```

Разделение endpoints на зоны ученика и учителя позволяет четко разграничить функциональность и права доступа. Каждый endpoint автоматически получает стандартный набор операций (создание, чтение, обновление, удаление), а также может содержать дополнительные действия для специфических операций, таких как проверка ответов на этапе осмысления или сохранение схемы, созданной учеником.

Клиентская часть реализована на библиотеке React версии 18.x. Выбор этой библиотеки обусловлен её компонентной архитектурой, которая позволяет создавать переиспользуемые элементы интерфейса и эффективно управлять состоянием приложения. Для сборки проекта используется инструмент Vite, который обеспечивает быструю разработку с поддержкой горячей перезагрузки модулей при изменении кода. Стилизация интерфейса выполнена с помощью библиотеки Tailwind CSS, которая предоставляет набор утилитарных классов для описания внешнего вида элементов.

Интерфейс ученика и панель учителя реализованы как отдельные компоненты верхнего уровня. Это позволяет независимо развивать функциональность каждой части и не смешивать логику работы разных типов пользователей. Интерфейс ученика использует механизм контекста React для передачи данных между компонентами без явной передачи через свойства. Это упрощает структуру кода и позволяет дочерним компонентам получать доступ к общему состоянию приложения.

Главный компонент интерфейса ученика управляет переключением между двумя основными представлениями: каталогом систем знаний и процессом обучения. При выборе системы знаний происходит загрузка данных через API и создание или получение учебной сессии. Все данные передаются через контекст, что позволяет дочерним компонентам получать к ним доступ без явной передачи через свойства. В листинге 3.3 показан фрагмент главного компонента, демонстрирующий структуру приложения и использование контекста.

Листинг 3.3 – фрагмент главного компонента интерфейса ученика

```jsx
export default function App() {
  const [catalog, setCatalog] = useState([]);
  const [selectedKS, setSelectedKS] = useState(null);
  const [ksData, setKsData] = useState(null);
  const [session, setSession] = useState(null);
  const [view, setView] = useState("catalog");

  useEffect(() => {
    const r = await fetch("/api/catalog/");
    const data = await r.json();
    setCatalog(data);
  }, []);

  const contextValue = {
    catalog,
    selectedKS,
    setSelectedKS,
    ksData,
    session,
    setSession,
  };

  return (
    <AppContext.Provider value={contextValue}>
      <div className="min-h-screen bg-slate-50">
        {view === "catalog" && <CatalogView />}
        {view === "learning" && ksData && <LearningView />}
      </div>
    </AppContext.Provider>
  );
}
```

Компонент использует условный рендеринг для переключения между каталогом и процессом обучения. При выборе системы знаний происходит загрузка данных через API и создание учебной сессии. Все данные передаются через контекст, что упрощает доступ к ним из дочерних компонентов.

Для управления этапами обучения реализован компонент, который выполняет роль маршрутизатора этапов. В зависимости от текущего этапа сессии отображается соответствующий компонент: осмысление системы знаний, формулировка типовой задачи, список задач, решение задач различной сложности и другие. Это позволяет реализовать последовательное прохождение этапов методики Гальперина: от ознакомления с системой знаний к её осмыслению и последующему решению задач.

Каждый этап обучения реализован в виде отдельного компонента. Например, компонент этапа осмысления отображает изображение таблицы с интерактивными зонами, вопросы различных типов и тексты с пропусками. Компонент управляет состоянием ответов ученика и отправляет их на сервер для проверки. После проверки отображается результат: процент правильных ответов, информация о достижении порога успешности и возможность повторной попытки.

Панель учителя реализована в отдельном компоненте. Она использует условный рендеринг для переключения между различными режимами работы: управление структурой курса, редактирование системы знаний, управление задачами и элементами схем. Это позволяет учителю последовательно работать с различными аспектами создания учебного контента, не перегружая интерфейс избыточными элементами.

Для работы с графическими схемами «модели ситуации» используется компонент редактора схем, который загружается динамически с помощью механизма ленивой загрузки React. Это позволяет уменьшить начальный размер загружаемого кода и ускорить первоначальную загрузку приложения. Компонент построен на библиотеке Konva.js, которая работает поверх холста HTML5 и предоставляет инструменты для создания и редактирования векторной графики.

Таким образом, архитектура электронного образовательного ресурса строится вокруг нескольких устойчивых блоков: структуры курса, подсистемы осмысления, подсистемы решения задач и подсистемы фиксации прогресса ученика. Связь между ними обеспечивается через единый API и общую модель данных. Такой подход позволил реализовать методику П. Я. Гальперина не только на уровне сценарного описания, но и в программной архитектуре: каждый этап обучения опирается на предыдущий, а данные о действиях ученика сохраняются и могут использоваться для адаптации последующих заданий.
