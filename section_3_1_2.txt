### 3.1.2. Используемые программные средства и технологии разработки

При выборе программных средств для разработки электронного образовательного ресурса автор исходил из необходимости совместить гибкость разработки, удобство для последующей доработки и достаточную производительность системы. Также учитывалась необходимость обеспечения современного пользовательского интерфейса и возможности работы с графическими элементами.

Серверная часть реализована на языке Python версии 3.12 с использованием веб-фреймворка Django версии 4.x. Выбор Django обусловлен его развитой системой работы с базами данных через объектно-реляционное отображение, встроенной системой аутентификации и возможностью быстрого создания административного интерфейса для управления контентом. Для организации взаимодействия с клиентской частью используется подсистема Django REST Framework, которая позволяет описывать прикладной программный интерфейс в виде набора представлений и сериализаторов. Это упрощает создание API и обеспечивает единообразный формат обмена данными.

На этапе разработки в качестве системы управления базами данных применялась SQLite. Это упростило развёртывание проекта и позволило сосредоточиться на логике работы приложения без необходимости настройки отдельного сервера базы данных. При переносе в эксплуатационную среду возможно использование PostgreSQL без изменения логики работы приложения, так как доступ к данным организован через ORM Django, который абстрагирует различия между системами управления базами данных.

Для обработки межсайтовых запросов используется библиотека django-cors-headers. Она позволяет клиентской части, работающей на другом порту в режиме разработки, обращаться к серверной части без ограничений политики безопасности браузера. В настройках Django также включена поддержка работы с медиафайлами (изображения таблиц, схем, решений задач) и статическими файлами (скомпилированные JavaScript и CSS файлы клиентской части).

Клиентская часть реализована на библиотеке React версии 19.1. Такой выбор обусловлен развитой экосистемой библиотеки, поддержкой компонентного подхода и возможностью гибко организовать состояние приложения. React позволяет создавать переиспользуемые компоненты интерфейса и эффективно управлять обновлением пользовательского интерфейса при изменении данных. Для сборки и разработки используется инструмент Vite версии 7.1. Он обеспечивает быструю разработку с поддержкой горячей перезагрузки модулей при изменении кода, что значительно ускоряет процесс разработки интерфейса.

Для стилизации интерфейса применена библиотека Tailwind CSS версии 4.1. Она позволяет описывать внешний вид элементов с помощью утилитарных классов, что ускоряет верстку экранных форм и облегчает поддержание единого стиля во всей системе. Tailwind CSS предоставляет готовый набор классов для работы с цветами, отступами, типографикой и другими аспектами оформления, что избавляет от необходимости писать собственные стили для каждого элемента.

Отдельный блок функциональности связан с графическим редактором схем «модели ситуации». Для его реализации применена библиотека Konva.js версии 10.0, работающая поверх холста HTML5. На её основе создан компонент редактора, который позволяет добавлять объекты на холст, перемещать их, изменять размеры и связывать между собой. Важной особенностью является поддержка «привязки» объектов друг к другу и к сетке, а также наличие истории действий для операций «отменить» и «повторить». Это делает работу с редактором ближе к привычным для учителя инструментам и снижает порог входа.

Структура проекта организована таким образом, чтобы разделить серверную и клиентскую части. Серверная часть находится в директории `eora`, которая содержит приложение Django `learning` с моделями данных, представлениями и сериализаторами. Клиентская часть находится в директории `ui` и представляет собой отдельный проект на Vite. При сборке клиентская часть компилируется в статические файлы, которые размещаются в директории `static/app` серверной части. Это позволяет развернуть систему как единое веб-приложение, где Django обслуживает и статические файлы, и API.

В листинге 3.4 показан фрагмент конфигурации Vite, демонстрирующий настройку сборки клиентской части и размещение результата в директории статических файлов Django.

Листинг 3.4 – фрагмент конфигурации сборки клиентской части

```javascript
export default defineConfig({
  plugins: [react()],
  build: {
    outDir: '../static/app',
    emptyOutDir: true,
    rollupOptions: {
      input: {
        main: resolve(__dirname, 'index.html'),
        teacher: resolve(__dirname, 'teacher.html'),
      },
    },
  },
  base: '/static/app/',
})
```

Конфигурация определяет, что результат сборки должен быть размещён в директории `static/app` относительно корня проекта. Также настроена сборка двух точек входа: основной интерфейс ученика и панель учителя. Это позволяет создавать отдельные HTML-файлы для каждого интерфейса, что упрощает маршрутизацию на стороне сервера.

В листинге 3.5 показан фрагмент файла зависимостей клиентской части, демонстрирующий основные используемые библиотеки.

Листинг 3.5 – фрагмент зависимостей клиентской части

```json
{
  "dependencies": {
    "konva": "^10.0.12",
    "react": "^19.1.1",
    "react-dom": "^19.1.1",
    "react-konva": "^19.2.1"
  },
  "devDependencies": {
    "vite": "^7.1.7",
    "tailwindcss": "^4.1.14",
    "@vitejs/plugin-react": "^5.0.4"
  }
}
```

Основные зависимости включают React для построения интерфейса, Konva для работы с графикой и библиотеку react-konva, которая обеспечивает интеграцию Konva с React. В зависимостях для разработки находятся инструменты сборки и стилизации.

Всё приложение разворачивается как веб-сервис. Запросы от браузера передаются через стандартный протокол HTTP, обмен данными идёт в формате JSON. Учёт сессии пользователя организован на стороне сервера с использованием механизма сессий Django. Для защиты от межсайтовой подделки запросов используется токен CSRF, который передаётся в заголовках HTTP-запросов от клиента к серверу.

При разработке использовался специальный режим, в котором школьник создаётся автоматически при первом обращении к ресурсу. Это позволило сосредоточиться на отладке логики учебного процесса, а вопросы полноценной авторизации и разграничения прав доступа были вынесены на более поздний этап. В эксплуатационной версии системы предполагается использование стандартной системы аутентификации Django с поддержкой различных типов пользователей: учеников, учителей и администраторов.

Таким образом, выбор технологий обеспечил создание современного веб-приложения с разделением на серверную и клиентскую части, возможностью работы с графическими элементами и удобной системой разработки. Использование проверенных и широко распространённых библиотек гарантирует стабильность работы системы и возможность её дальнейшего развития.
