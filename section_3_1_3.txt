### 3.1.3. Структура базы данных и модели предметной области

Структура базы данных отражает логику построения учебного курса и реализуемых этапов обучения в соответствии с методикой П. Я. Гальперина. В основе лежит иерархия «класс – раздел – тема – система знаний», которая позволяет организовать учебный материал в структуру, близкую к структуре учебника и рабочей программы. Каждая система знаний содержит собственные средства для осмысления, метод решения задач и набор связанных заданий.

Иерархия курса реализована через четыре основные модели: `SchoolClass` (класс обучения), `SubjectSection` (раздел предмета), `Topic` (тема) и `KnowledgeSystem` (система знаний). Каждая модель связана с вышестоящей через внешний ключ, что обеспечивает каскадное удаление данных при удалении родительского элемента. Это гарантирует целостность данных и предотвращает появление «висячих» записей в базе данных. В листинге 3.6 показан фрагмент моделей иерархии курса, демонстрирующий структуру и связи.

Листинг 3.6 – фрагмент моделей иерархии курса

```python
class SchoolClass(models.Model):
    number = models.PositiveSmallIntegerField(unique=True)
    title = models.CharField(max_length=50)

class SubjectSection(models.Model):
    school_class = models.ForeignKey(SchoolClass, on_delete=models.CASCADE, 
                                     related_name="sections")
    title = models.CharField(max_length=200)
    order = models.PositiveIntegerField(default=1)

class Topic(models.Model):
    section = models.ForeignKey(SubjectSection, on_delete=models.CASCADE, 
                                related_name="topics")
    title = models.CharField(max_length=255)
    order = models.PositiveIntegerField(default=1)

class KnowledgeSystem(models.Model):
    topic = models.ForeignKey(Topic, on_delete=models.CASCADE, 
                              related_name="knowledge_systems")
    title = models.CharField(max_length=255)
    comprehension_image = models.ImageField(upload_to="ks/comprehension/",
                                           blank=True, null=True)
    comprehension_pass_threshold = models.PositiveIntegerField(default=85)
```

Модель класса содержит номер класса и его название. Модель раздела связана с классом и содержит название раздела и порядковый номер для сортировки. Модель темы связана с разделом и также содержит название и порядковый номер. Модель системы знаний является конечным элементом иерархии и содержит ссылку на тему, название системы знаний, изображение для этапа осмысления и порог успешности прохождения этого этапа.

Для реализации этапа осмысления системы знаний разработаны специализированные модели данных. Модель `KSZone` описывает прямоугольные зоны на изображении таблицы. Каждая зона содержит координаты верхнего левого угла (x, y), ширину и высоту прямоугольника. Модель `KSQuestion` описывает вопросы различных типов: открытый ответ, выбор одного варианта, множественный выбор и соответствие зон. Для вопросов с вариантами ответов используется поле типа JSON, которое хранит список вариантов с отметкой о правильности. Для вопросов типа «соответствие» используется связь «многие-ко-многим» с моделью зон, что позволяет указать, какие зоны должны быть выбраны для правильного ответа.

Модель `KSCloze` предназначена для текстов с пропусками. Она содержит исходный текст, текст с маркерами пропусков (в формате `{{0}}`, `{{1}}` и т.д.), структуру пропусков в формате JSON и список отвлекающих вариантов. Структура пропусков хранит для каждого пропуска его позицию в тексте и правильный ответ. Это позволяет динамически формировать варианты ответов для каждого пропуска, объединяя правильный ответ с отвлекающими вариантами и перемешивая их для ученика.

Для фиксации результатов осмысления используются модели `ComprehensionAttempt`, `QuestionAnswer` и `ClozeAnswer`. Модель `ComprehensionAttempt` связывает попытку осмысления с учебной сессией и хранит общий результат: количество вопросов, количество правильных ответов, процент правильных ответов и флаг успешного прохождения. Модели `QuestionAnswer` и `ClozeAnswer` хранят ответы ученика на конкретные вопросы и тексты с пропусками соответственно. Они содержат ссылки на попытку осмысления и на сам вопрос или текст с пропусками, а также результат проверки ответа.

Для реализации метода решения задач используются модели `SolutionMethod` и `SolutionStep`. Модель `SolutionMethod` связана с системой знаний отношением «один-к-одному», что означает, что каждая система знаний имеет ровно один метод решения. Модель `SolutionStep` описывает отдельный шаг метода и связана с методом отношением «один-ко-многим». Каждый шаг содержит порядковый номер, название действия, описание и подсказку для ученика. Это позволяет структурировать метод решения как последовательность из десяти шагов, что соответствует методике Гальперина.

Для задач реализована модель `Task`, которая связана с системой знаний отношением «один-ко-многим». Модель содержит текст условия задачи, правильный ответ (числовой или текстовый), единицу измерения, допустимую погрешность и уровень сложности. Также в модели хранятся эталонные решения: краткое решение, развёрнутое решение и изображение с решением. Модель `TaskSolutionStep` связывает задачу с шагами метода решения и содержит результат выполнения каждого шага для конкретной задачи. Это позволяет показать ученику, как метод решения применяется к конкретной задаче.

Для фиксации хода обучения введены модели `LearningSession`, `TaskAttempt` и `StepAttempt`. Модель `LearningSession` является центральной для отслеживания прогресса ученика. Она связывает пользователя с системой знаний и хранит текущий этап обучения, статистику выполнения заданий и общий результат. Модель `TaskAttempt` описывает попытку решения конкретной задачи и содержит ответ ученика, результат проверки, схему, созданную учеником, и оценку сложности задачи учеником. Модель `StepAttempt` описывает попытку выполнения отдельного шага метода решения и используется для пооперационного контроля.

Для работы с графическими схемами «модели ситуации» реализованы модели `SchemaElementCategory`, `SchemaElement`, `SchemaTemplate` и `StudentSchema`. Модель `SchemaElementCategory` описывает категории элементов схемы для удобной классификации. Модель `SchemaElement` описывает отдельный элемент схемы и содержит SVG-код для его отображения, свойства по умолчанию и список редактируемых свойств. Модель `SchemaTemplate` описывает шаблон схемы для задачи и может быть эталонной схемой, начальной заготовкой или подсказкой. Модель `StudentSchema` описывает схему, созданную учеником, и содержит оценку похожести на эталон и обратную связь.

Схема связей между основными сущностями представлена на рисунке 3.1 в виде диаграммы «сущность – связь». В центре диаграммы располагаются модели `Topic` и `KnowledgeSystem`, к которым «подвешены» как задания, так и элементы осмысления. Модели `KSZone`, `KSQuestion` и `KSCloze` связаны с системой знаний отношениями «один-ко-многим». Модели `LearningSession`, `ComprehensionAttempt` и попытки решения задач связаны с пользователем и соответствующими учебными объектами. Такое построение позволяет, с одной стороны, хранить результаты по каждому этапу отдельно, а с другой – при необходимости агрегировать данные и строить общую картину сформированности действий по методике Гальперина.

За счёт использования ORM Django логическая структура предметной области непосредственно задаёт структуру базы данных. Определения моделей в коде одновременно являются и документацией по данным. Это облегчает сопровождение проекта: при изменении требований к учебному процессу возможно внесение точечных корректировок в модели и проведение миграций, без ручного изменения таблиц в базе. Django автоматически генерирует SQL-запросы для создания и изменения структуры базы данных на основе определений моделей, что гарантирует соответствие структуры базы данных логической модели предметной области.
