## 1. Введение: зачем нужны модели данных

### 1.1. Проблема без моделей

Когда мы создаём веб-приложение, нам нужно хранить данные:
- Информацию о пользователях
- Учебные материалы
- Результаты обучения
- Настройки системы

**Без моделей** мы бы писали SQL-запросы вручную:
```sql
CREATE TABLE knowledge_systems (
    id INTEGER PRIMARY KEY,
    title VARCHAR(255),
    description TEXT,
    created_at DATETIME
);
```

**Проблемы:**
- Много повторяющегося кода
- Сложно поддерживать
- Легко допустить ошибки
- Нет типизации

### 1.2. Решение: Django ORM (Object-Relational Mapping)

**ORM** — это техника, которая позволяет работать с базой данных через объекты Python, а не через SQL.

**Преимущества:**
- ✅ Код на Python (не SQL)
- ✅ Автоматическая типизация
- ✅ Встроенная валидация
- ✅ Миграции (автоматическое обновление БД)
- ✅ Защита от SQL-инъекций

**Пример:**
```python
# Вместо SQL:
# SELECT * FROM knowledge_systems WHERE title LIKE '%движение%';

# Пишем на Python:
systems = KnowledgeSystem.objects.filter(title__icontains='движение')
```

## 2. Основы Django ORM

### 2.1. Структура модели

Модель в Django — это класс, который наследуется от `models.Model`.

**Базовый пример:**
```python
from django.db import models

class KnowledgeSystem(models.Model):
    title = models.CharField(max_length=255)
    description = models.TextField(blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
```

**Что происходит:**
1. Django автоматически создаёт таблицу в БД
2. Каждое поле становится колонкой в таблице
3. Django добавляет поле `id` (первичный ключ) автоматически

### 2.2. Где создавать модели?

Модели создаются в файле `models.py` внутри Django-приложения:

```
eora/
├── learning/
│   ├── models.py  ← здесь создаём модели
│   ├── views.py
│   └── ...
```

**Структура файла `models.py`:**
```python
from django.db import models

# =============================================================================
# 1. УЧЕБНАЯ ИЕРАРХИЯ
# =============================================================================

class SchoolClass(models.Model):
    """Класс обучения (7, 8, 9)"""
    number = models.PositiveSmallIntegerField(unique=True)
    title = models.CharField(max_length=50)

    class Meta:
        verbose_name = "Класс"
        verbose_name_plural = "Классы"
        ordering = ["number"]

    def __str__(self):
        return self.title
```

**Важно:**
- Используйте комментарии для группировки моделей
- Добавляйте docstring (тройные кавычки) для описания
- Метод `__str__()` определяет, как модель отображается в админке

## 3. Типы полей в Django

### 3.1. Текстовые поля

**CharField** — для коротких строк (до 255 символов):
```python
title = models.CharField(max_length=255)
```

**TextField** — для длинных текстов:
```python
description = models.TextField(blank=True)
```

**Параметры:**
- `max_length` — максимальная длина (обязательно для CharField)
- `blank=True` — поле может быть пустым в формах
- `null=True` — поле может быть NULL в БД (для TextField обычно не нужно)

### 3.2. Числовые поля

**IntegerField** — целое число:
```python
number = models.IntegerField()
```

**PositiveIntegerField** — положительное целое:
```python
order = models.PositiveIntegerField(default=1)
```

**FloatField** — число с плавающей точкой:
```python
score = models.FloatField(default=0.0)
```

**PositiveSmallIntegerField** — маленькое положительное (до 32767):
```python
difficulty = models.PositiveSmallIntegerField(choices=DIFFICULTY_CHOICES)
```

### 3.3. Логические поля

**BooleanField** — да/нет:
```python
is_published = models.BooleanField(default=False)
```

### 3.4. Поля даты и времени

**DateTimeField** — дата и время:
```python
created_at = models.DateTimeField(auto_now_add=True)  # Устанавливается при создании
updated_at = models.DateTimeField(auto_now=True)      # Обновляется при каждом сохранении
```

**DateField** — только дата:
```python
birth_date = models.DateField()
```

### 3.5. Поля для файлов

**ImageField** — для изображений:
```python
image = models.ImageField(
    upload_to="ks/",  # Куда сохранять файлы
    blank=True,
    null=True
)
```

**FileField** — для любых файлов:
```python
document = models.FileField(upload_to="documents/")
```

**Важно:** Для работы с изображениями нужно установить `Pillow`:
```bash
pip install Pillow
```

### 3.6. Поля с выбором (choices)

Когда поле может иметь только определённые значения:

```python
class KnowledgeSystem(models.Model):
    STATUS_CHOICES = [
        ("draft", "Черновик"),
        ("published", "Опубликовано"),
    ]
    
    status = models.CharField(
        max_length=10,
        choices=STATUS_CHOICES,
        default="draft"
    )
```

**Использование:**
```python
system = KnowledgeSystem.objects.get(id=1)
print(system.status)  # "draft"
print(system.get_status_display())  # "Черновик"
```

### 3.7. Параметры полей (общие)

**default** — значение по умолчанию:
```python
order = models.PositiveIntegerField(default=1)
```

**unique** — уникальное значение:
```python
number = models.PositiveSmallIntegerField(unique=True)
```

**help_text** — подсказка (для админки):
```python
title = models.CharField(
    max_length=255,
    help_text="Название системы знаний"
)
```

## 4. Связи между моделями

В реальных приложениях модели связаны друг с другом. Django поддерживает три типа связей.

### 4.1. ForeignKey (один-ко-многим)

**Пример:** Одна тема может иметь много систем знаний, но система знаний принадлежит одной теме.

```python
class Topic(models.Model):
    title = models.CharField(max_length=255)

class KnowledgeSystem(models.Model):
    topic = models.ForeignKey(
        Topic,
        on_delete=models.CASCADE,
        related_name="knowledge_systems"
    )
    title = models.CharField(max_length=255)
```

**Параметры:**
- `on_delete=models.CASCADE` — при удалении темы удаляются все связанные системы знаний
- `related_name="knowledge_systems"` — позволяет обращаться к системам знаний через тему: `topic.knowledge_systems.all()`

**Использование:**
```python
# Получить все системы знаний для темы
topic = Topic.objects.get(id=1)
systems = topic.knowledge_systems.all()

# Получить тему системы знаний
system = KnowledgeSystem.objects.get(id=1)
print(system.topic.title)
```

**Другие варианты `on_delete`:**
- `models.PROTECT` — запрещает удаление, если есть связанные объекты
- `models.SET_NULL` — устанавливает NULL (нужно `null=True`)
- `models.SET_DEFAULT` — устанавливает значение по умолчанию

### 4.2. OneToOneField (один-к-одному)

**Пример:** У каждой системы знаний может быть один метод решения, и метод решения принадлежит одной системе знаний.

```python
class KnowledgeSystem(models.Model):
    title = models.CharField(max_length=255)

class SolutionMethod(models.Model):
    ks = models.OneToOneField(
        KnowledgeSystem,
        on_delete=models.CASCADE,
        related_name="solution_method"
    )
    title = models.CharField(max_length=255)
```

**Использование:**
```python
system = KnowledgeSystem.objects.get(id=1)
method = system.solution_method  # Получить метод решения

method = SolutionMethod.objects.get(id=1)
system = method.ks  # Получить систему знаний
```

### 4.3. ManyToManyField (многие-ко-многим)

**Пример:** Вопрос может быть связан с несколькими зонами, и зона может быть связана с несколькими вопросами.

```python
class KSZone(models.Model):
    x = models.PositiveIntegerField()
    y = models.PositiveIntegerField()

class KSQuestion(models.Model):
    text = models.TextField()
    zones = models.ManyToManyField(
        KSZone,
        related_name="questions",
        blank=True
    )
    correct_zones = models.ManyToManyField(
        KSZone,
        related_name="correct_for_questions",
        blank=True
    )
```

**Использование:**
```python
question = KSQuestion.objects.get(id=1)
zones = question.zones.all()  # Все зоны вопроса

zone = KSZone.objects.get(id=1)
questions = zone.questions.all()  # Все вопросы с этой зоной

# Добавить связь
question.zones.add(zone)

# Удалить связь
question.zones.remove(zone)
```

**Важно:** Django автоматически создаёт промежуточную таблицу для ManyToMany.

### 4.4. Пример из реального проекта

Рассмотрим иерархию учебных материалов:

```python
# =============================================================================
# 1. УЧЕБНАЯ ИЕРАРХИЯ
# =============================================================================

class SchoolClass(models.Model):
    """Класс обучения (7, 8, 9)"""
    number = models.PositiveSmallIntegerField(unique=True)
    title = models.CharField(max_length=50)

class SubjectSection(models.Model):
    """Раздел предмета (например, 'Механические явления')"""
    school_class = models.ForeignKey(
        SchoolClass,
        on_delete=models.CASCADE,
        related_name="sections"
    )
    title = models.CharField(max_length=200)
    order = models.PositiveIntegerField(default=1)

class Topic(models.Model):
    """Тема (например, 'Равномерное и неравномерное движение')"""
    section = models.ForeignKey(
        SubjectSection,
        on_delete=models.CASCADE,
        related_name="topics"
    )
    title = models.CharField(max_length=255)
    order = models.PositiveIntegerField(default=1)

class KnowledgeSystem(models.Model):
    """Система знаний — ядро обучения"""
    topic = models.ForeignKey(
        Topic,
        on_delete=models.CASCADE,
        related_name="knowledge_systems"
    )
    title = models.CharField(max_length=255)
    description = models.TextField(blank=True)
```

**Структура:**
```
Класс (7 класс)
  └── Раздел (Механические явления)
      └── Тема (Равномерное движение)
          └── Система знаний (Равномерное и неравномерное движение)
```

**Запросы:**
```python
# Получить все системы знаний для 7 класса
class_7 = SchoolClass.objects.get(number=7)
systems = KnowledgeSystem.objects.filter(
    topic__section__school_class=class_7
)

# Получить все темы раздела
section = SubjectSection.objects.get(id=1)
topics = section.topics.all()
```

## 5. Мета-классы и настройки моделей

Класс `Meta` внутри модели позволяет настроить поведение модели.

### 5.1. Основные настройки

```python
class KnowledgeSystem(models.Model):
    title = models.CharField(max_length=255)
    
    class Meta:
        verbose_name = "Система знаний"  # Единственное число (для админки)
        verbose_name_plural = "Системы знаний"  # Множественное число
        ordering = ["-created_at"]  # Сортировка по умолчанию
        # ordering = ["title"]  # По алфавиту
        # ordering = ["-created_at", "title"]  # Сначала новые, потом по алфавиту
```

**Параметры:**
- `verbose_name` — название в единственном числе
- `verbose_name_plural` — название во множественном числе
- `ordering` — порядок сортировки по умолчанию
  - `"-created_at"` — от новых к старым
  - `"created_at"` — от старых к новым

### 5.2. Индексы для производительности

Если часто ищем по определённому полю, можно добавить индекс:

```python
class Task(models.Model):
    ks = models.ForeignKey(KnowledgeSystem, on_delete=models.CASCADE)
    difficulty = models.PositiveSmallIntegerField()
    
    class Meta:
        indexes = [
            models.Index(fields=["ks", "difficulty"]),  # Составной индекс
        ]
```

## 6. JSONField для гибких данных

Иногда структура данных может быть разной для разных объектов. Вместо создания множества полей можно использовать `JSONField`.

### 6.1. Пример: варианты ответов в вопросах

**Проблема:** У разных типов вопросов разные структуры данных:
- Вопрос с выбором: `["Вариант 1", "Вариант 2", ...]`
- Вопрос с соответствием: `[{"zone_id": 1, "correct": true}, ...]`

**Решение:** Использовать JSONField:

```python
class KSQuestion(models.Model):
    TYPE_CHOICES = [
        ("single", "Выбор одного варианта"),
        ("multiple", "Множественный выбор"),
        ("match", "Соответствие"),
    ]
    
    type = models.CharField(max_length=20, choices=TYPE_CHOICES)
    text = models.TextField()
    
    # Варианты ответа хранятся в JSON
    options = models.JSONField(
        default=list,
        blank=True,
        help_text='[{"text": "Вариант 1", "is_correct": false}, ...]'
    )
```

**Использование:**
```python
# Создать вопрос
question = KSQuestion.objects.create(
    type="single",
    text="Что такое равномерное движение?",
    options=[
        {"text": "Движение с постоянной скоростью", "is_correct": True},
        {"text": "Движение с ускорением", "is_correct": False},
        {"text": "Движение по окружности", "is_correct": False},
    ]
)

# Получить варианты
for option in question.options:
    print(option["text"], option["is_correct"])
```

### 6.2. Пример: пропуски в тексте (Cloze)

```python
class KSCloze(models.Model):
    """Текст с пропусками"""
    marked_text = models.TextField(
        help_text='Текст с маркерами {{0}}, {{1}}, ...'

    # Пропуски хранятся в JSON

        help_text='["неверное1", "неверное2", ...]'

```

**Пример данных:**
```python
cloze = KSCloze.objects.create(
    marked_text="Это {{0}} движение со скоростью {{1}} м/с",
    blanks=[
        {"position": 0, "correct": "равномерное"},
        {"position": 1, "correct": "5"},

    distractors=["неравномерное", "10", "15"]
)
```

### 6.3. Методы для работы с JSONField

Можно добавить методы модели для удобной работы:

```python
class KSCloze(models.Model):
    blanks = models.JSONField(default=list)
    distractors = models.JSONField(default=list)
    
    def get_all_options(self):
        """Получить все варианты (правильные + отвлекатели) в случайном порядке"""
        import random
        correct_words = [b["correct"] for b in self.blanks]
        all_options = list(set(correct_words + self.distractors))
        random.shuffle(all_options)
        return all_options
    
    def check_answers(self, answers):
        """
        Проверка ответов ученика.
        answers: {0: "слово1", 1: "слово2", ...}
        Возвращает: {"correct": [0, 2], "wrong": [1], "score": 0.66}
        """
        correct = []
        wrong = []
        
        for blank in self.blanks:
            pos = blank["position"]
            student_answer = answers.get(pos, "").strip().lower()
            correct_answer = blank["correct"].strip().lower()
            
            if student_answer == correct_answer:
                correct.append(pos)
            else:
                wrong.append(pos)
        
        total = len(self.blanks)
        score = len(correct) / total if total > 0 else 0
        
        return {
            "correct": correct,
            "wrong": wrong,
            "score": round(score, 2)
        }
```

**Использование:**
```python
cloze = KSCloze.objects.get(id=1)
options = cloze.get_all_options()  # ["равномерное", "неравномерное", "5", ...]

student_answers = {0: "равномерное", 1: "5"}
result = cloze.check_answers(student_answers)
# {"correct": [0, 1], "wrong": [], "score": 1.0}
```

**Преимущества JSONField:**
- ✅ Гибкость — можно хранить любую структуру
- ✅ Не нужно создавать дополнительные таблицы
- ✅ Легко добавлять новые поля

**Недостатки:**
- ❌ Сложнее делать запросы (нужны специальные функции)
- ❌ Нет валидации структуры на уровне БД
- ❌ Может быть медленнее для больших объёмов данных

## 7. Методы моделей и бизнес-логика

Модели могут содержать не только данные, но и методы для работы с ними.

### 7.1. Метод `__str__()`

Определяет, как объект отображается в строках и админке:

```python
class KnowledgeSystem(models.Model):
    title = models.CharField(max_length=255)
    version = models.PositiveIntegerField(default=1)
    
    def __str__(self):
        return f"{self.title} v{self.version}"
```

**Результат:**
```python
system = KnowledgeSystem.objects.get(id=1)
print(system)  # "Равномерное движение v1"
```

### 7.2. Методы для бизнес-логики

**Пример: проверка ответа на вопрос**

```python
class KSQuestion(models.Model):
    type = models.CharField(max_length=20, choices=TYPE_CHOICES)
    text = models.TextField()
    options = models.JSONField(default=list)
    correct_answer_text = models.CharField(max_length=500, blank=True)
    
    def check_answer(self, answer):
        """
        Проверка ответа ученика.
        answer: str (для text), int (для single), list[int] (для multiple)
        """
        if self.type == "text":
            return answer.strip() == self.correct_answer_text.strip()
        
        elif self.type == "single":
            # answer — индекс выбранного варианта
            if 0 <= answer < len(self.options):
                return self.options[answer].get("is_correct", False)
            return False
        
        elif self.type == "multiple":
            # answer — список индексов
            correct_indices = {
                i for i, opt in enumerate(self.options)
                if opt.get("is_correct")
            }
            return set(answer) == correct_indices
        
        return False
```

**Использование:**
```python
question = KSQuestion.objects.get(id=1)

# Для вопроса с выбором
is_correct = question.check_answer(0)  # Выбран первый вариант

# Для вопроса с множественным выбором
is_correct = question.check_answer([0, 2])  # Выбраны первый и третий варианты
```

### 7.3. Методы для вычисляемых полей

Иногда нужно вычислить значение на основе других полей:

```python
class TaskAttempt(models.Model):
    """Попытка решения задачи"""
    task = models.ForeignKey(Task, on_delete=models.CASCADE)
    student_answer = models.FloatField()
    is_correct = models.BooleanField(default=False)
    time_spent = models.PositiveIntegerField()  # секунды
    
    def get_score(self):
        """Вычислить балл за попытку"""
        if self.is_correct:
            # Бонус за скорость (максимум 10 баллов)
            if self.time_spent < 60:
                return 10
            elif self.time_spent < 120:
                return 8
            else:
                return 5
        return 0
```

## 8. Миграции базы данных

Когда вы создаёте или изменяете модели, Django создаёт **миграции** — файлы, которые описывают изменения в структуре БД.

### 8.1. Создание миграций

После изменения моделей:

```bash
python manage.py makemigrations
```

**Что происходит:**
1. Django сравнивает текущие модели с предыдущими миграциями
2. Создаёт файл миграции в папке `migrations/`
3. Файл содержит инструкции для изменения БД

**Пример файла миграции:**
```python
# learning/migrations/0001_initial.py
from django.db import migrations, models

class Migration(migrations.Migration):
    initial = True
    
    dependencies = []
    
    operations = [
        migrations.CreateModel(
            name='KnowledgeSystem',
            fields=[
                ('id', models.AutoField(primary_key=True)),
                ('title', models.CharField(max_length=255)),
                ('description', models.TextField(blank=True)),
            ],
        ),
    ]
```

### 8.2. Применение миграций

Чтобы применить изменения к БД:

```bash
python manage.py migrate
```

**Что происходит:**
1. Django выполняет все неприменённые миграции
2. Создаёт/изменяет таблицы в БД
3. Помечает миграции как применённые

### 8.3. Откат миграций

Если нужно откатить последнюю миграцию:

```bash
python manage.py migrate learning 0001  # Откатить до миграции 0001
```

### 8.4. Просмотр SQL-запросов

Чтобы увидеть, какие SQL-запросы выполнит миграция:

```bash
python manage.py sqlmigrate learning 0001
```

### 8.5. Важные правила

1. **Не редактируйте файлы миграций вручную** (кроме особых случаев)
2. **Коммитьте миграции в Git** — они часть кода
3. **Применяйте миграции на всех серверах** (dev, staging, production)
4. **Делайте резервные копии БД** перед применением миграций на production

## Практические рекомендации

### 1. Именование моделей

- Используйте **единственное число**: `KnowledgeSystem`, а не `KnowledgeSystems`
- Названия должны быть **понятными** и **описательными**

### 2. Группировка моделей

Разделяйте модели по логическим блокам комментариями:

```python
# =============================================================================
# 1. УЧЕБНАЯ ИЕРАРХИЯ
# =============================================================================
class SchoolClass(models.Model): ...

# =============================================================================
# 2. СИСТЕМА ЗНАНИЙ
# =============================================================================
class KnowledgeSystem(models.Model): ...
```

### 3. Документация

Всегда добавляйте docstring к моделям:

```python
class KnowledgeSystem(models.Model):
    """
    Система знаний — ядро обучения.
    Содержит изображение-таблицу, вопросы для осмысления, метод решения, задачи.
    """
    ...
```

### 4. Использование related_name

Всегда указывайте `related_name` для ForeignKey и ManyToMany:

```python
topic = models.ForeignKey(
    Topic,
    on_delete=models.CASCADE,
    related_name="knowledge_systems"  # ← важно!
)
```

### 5. Производительность

Используйте `select_related()` и `prefetch_related()` для оптимизации запросов:

```python
# Плохо: N+1 запросов
systems = KnowledgeSystem.objects.all()
for system in systems:
    print(system.topic.title)  # Отдельный запрос для каждой системы

# Хорошо: 1 запрос
systems = KnowledgeSystem.objects.select_related('topic').all()
for system in systems:
    print(system.topic.title)  # Данные уже загружены
```

## Итоги

1. **Модели Django** — это Python-классы, которые описывают структуру БД
2. **Типы полей** — CharField, TextField, IntegerField, DateTimeField, ImageField и др.
3. **Связи** — ForeignKey (один-ко-многим), OneToOneField (один-к-одному), ManyToManyField (многие-ко-многим)
4. **JSONField** — для гибких структур данных
5. **Методы моделей** — для бизнес-логики и вычислений
6. **Миграции** — автоматическое управление изменениями БД

**Следующий шаг:** После создания моделей нужно создать API для работы с ними (Django REST Framework).
