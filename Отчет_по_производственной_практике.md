## 1. ВВЕДЕНИЕ

В рамках производственной практики была выполнена практическая реализация электронного образовательного ресурса (ЭОР) EORA (Educational Online Resource for Adaptive learning) — системы для адаптивного обучения физике в основной школе. Система разработана на основе методологии поэтапного формирования умственных действий и реализует адаптивный алгоритм выбора заданий в зависимости от успешности ученика.

Целью практики являлась реализация всех модулей системы, описанных в третьей главе магистерской диссертации, включая backend на Django, frontend на React, интеграцию компонентов и тестирование функциональности.

---

## 2. АРХИТЕКТУРА СИСТЕМЫ

Система EORA построена по архитектуре клиент-сервер с разделением на backend и frontend компоненты. Backend реализован на Django с использованием Django REST Framework для предоставления API. Frontend реализован на React с использованием современных библиотек и инструментов разработки.

Система состоит из двух основных интерфейсов: интерфейса учителя для управления контентом и интерфейса ученика для прохождения обучения. Оба интерфейса взаимодействуют с единым backend через REST API.

## 3. ОПИСАНИЕ МОДУЛЕЙ СИСТЕМЫ

### 3.1. Модуль управления учебной иерархией

Модуль управления учебной иерархией обеспечивает структурирование учебного материала по принципу "Класс → Раздел → Тема → Система знаний". Данный модуль реализует трехуровневую иерархию, которая позволяет учителям организовать контент в соответствии с учебной программой.

**Функциональность модуля:**

Модуль предоставляет возможность создания и управления классами обучения (7, 8, 9 классы), разделами предмета (например, "Механические явления") и темами (например, "Равномерное и неравномерное движение"). Каждый уровень иерархии связан с предыдущим через внешние ключи, что обеспечивает целостность данных и возможность каскадного удаления.

В интерфейсе учителя модуль представлен в виде дерева навигации, где учитель может создавать, редактировать и удалять элементы иерархии. Интерфейс ученика отображает каталог в виде структурированного списка, позволяющего выбрать систему знаний для изучения.

**API эндпоинты:**

Модуль предоставляет эндпоинт `/api/catalog/` для получения полной иерархии классов, разделов и тем с предзагруженными связанными системами знаний. Данные возвращаются в формате JSON с вложенной структурой, оптимизированной для отображения в интерфейсе.

### 3.2. Модуль систем знаний

Модуль систем знаний является центральным компонентом системы, так как система знаний представляет собой основную единицу обучения. Каждая система знаний содержит таблицу с физическими величинами и их взаимосвязями, которая служит основой для всех этапов обучения.

**Функциональность модуля:**

Модуль обеспечивает создание и управление системами знаний, включая загрузку изображений таблиц, настройку параметров прохождения (порог прохождения осмысления), формулировку типовой задачи и описание системы. Каждая система знаний связана с темой через внешний ключ и может находиться в статусе "черновик" или "опубликовано".

Система знаний содержит два типа изображений: основное изображение для отображения в каталоге и изображение для этапа осмысления с нанесенными зонами для интерактивного взаимодействия. Модуль поддерживает настройку отображения зон по умолчанию и установку порога прохождения этапа осмысления.

**API эндпоинты:**

Модуль предоставляет эндпоинты для получения детальной информации о системе знаний (`/api/ks/<id>/`), включая связанные зоны, вопросы, тексты с пропусками, метод решения и задачи. Для учителя доступны эндпоинты для создания, редактирования и удаления систем знаний.

### 3.3. Модуль осмысления системы знаний

Модуль осмысления реализует первый этап обучения, на котором ученик должен понять структуру системы знаний через интерактивное взаимодействие с изображением таблицы. Модуль поддерживает три типа заданий: сопоставление вопросов с зонами на изображении, заполнение пропусков в тексте (cloze) и выбор правильных вариантов из списка.

**Функциональность модуля:**

Модуль позволяет учителю создавать зоны на изображении системы знаний, которые ученик должен сопоставить с вопросами. Зоны определяются координатами прямоугольников на изображении и могут быть видимыми или скрытыми по умолчанию. Вопросы создаются учителем и связываются с одной или несколькими правильными зонами.

Для текстовых заданий модуль поддерживает создание текстов с пропусками (cloze), где каждое пропущенное слово хранится в JSON формате с указанием позиции и правильного варианта. Модуль также поддерживает добавление отвлекающих вариантов для увеличения сложности задания.

Проверка ответов ученика осуществляется автоматически: для вопросов проверяется соответствие выбранных зон правильным зонам, для текстов с пропусками — правильность заполнения каждого пропуска. Результат проверки выражается в процентах правильных ответов, и ученик проходит этап, если процент превышает установленный порог (по умолчанию 85%).

**API эндпоинты:**

Модуль предоставляет эндпоинты для получения данных осмысления (`/api/ks/<id>/comprehension/`), проверки ответов (`/api/ks/<id>/check/`) и управления контентом через панель учителя. Эндпоинт проверки принимает ответы ученика в формате JSON и возвращает детальную информацию о правильности каждого ответа.

### 3.4. Модуль задач

Модуль задач обеспечивает создание и управление задачами для применения системы знаний. Каждая задача связана с системой знаний и содержит условие, правильный ответ, эталонное решение и уровень сложности.

**Функциональность модуля:**

Модуль поддерживает создание задач с числовыми и текстовыми ответами. Для числовых ответов предусмотрена проверка с учетом допустимой погрешности, что позволяет учитывать округления и приблизительные вычисления учеников. Каждая задача имеет уровень сложности от 1 (очень легкая) до 5 (очень сложная), что используется адаптивным алгоритмом для выбора заданий.

Эталонное решение задачи может быть представлено в трех форматах: краткое решение (формула и ответ), развернутое решение (подробное описание) и изображение с решением. Кроме того, модуль поддерживает пошаговые решения задач, где каждый шаг соответствует шагу метода решения и может быть текстовым или представлять собой схему.

**API эндпоинты:**

Модуль предоставляет эндпоинты для получения списка задач системы знаний (`/api/ks/<id>/tasks/`), получения детальной информации о задаче (`/api/task/<id>/`), отправки ответа ученика (`/api/task/<id>/submit/`) и получения эталонного решения (`/api/task/<id>/solution/`). Для учителя доступны эндпоинты для управления задачами через панель учителя.

### 3.5. Модуль метода решения

Модуль метода решения реализует общий алгоритм решения задач для системы знаний, состоящий из последовательности шагов (обычно 10 шагов). Метод решения является универсальным для всех задач системы знаний и служит основой для этапов "Составление метода" и "Пооперационный контроль".

**Функциональность модуля:**

Модуль позволяет учителю создать метод решения, состоящий из последовательных шагов. Каждый шаг имеет порядковый номер, название, описание и подсказку для ученика. Учитель может настроить, какие шаги должны быть скрыты в этапе составления метода, чтобы ученик сам заполнил их названия.

Для каждой задачи учитель создает пошаговое решение, где каждый шаг метода заполняется конкретным содержанием для данной задачи. Шаг может быть текстовым (содержит текст решения) или представлять собой схему (содержит данные схемы в формате JSON). Модуль поддерживает наследование схемы от предыдущего шага, что позволяет ученику постепенно дополнять схему.

**API эндпоинты:**

Модуль предоставляет эндпоинты для получения метода решения (`/api/ks/<id>/method/`), проверки составленного метода (`/api/ks/<id>/check_method_composition/`), управления шагами метода и пошаговыми решениями задач через панель учителя.

### 3.6. Модуль сессий обучения и адаптивного алгоритма

Модуль сессий обучения отслеживает прогресс ученика при работе с системой знаний и реализует адаптивный алгоритм выбора заданий. Модуль является ключевым для персонализации обучения и обеспечения адаптивности системы.

**Функциональность модуля:**

Модуль создает и управляет сессиями обучения, каждая из которых связана с пользователем и системой знаний. Сессия отслеживает текущий этап обучения, количество решенных задач, количество правильных ответов, последовательность ошибок и общий процент успешности.

Адаптивный алгоритм реализован на основе нескольких факторов: выбора сложности учеником в начале обучения, последовательности правильных и неправильных ответов, общего процента правильных ответов. При двух ошибках подряд в режиме "легко" система автоматически переключает ученика на более сложный режим. Алгоритм также учитывает процент правильных ответов для корректировки сложности заданий.

Модуль логирует все события обучения (создание сессии, отправка ответов, переходы между этапами) для последующего анализа эффективности обучения.

**API эндпоинты:**

Модуль предоставляет эндпоинты для получения текущей сессии (`/api/session/current/`), создания новой сессии (`/api/session/start_new/`), установки выбора сложности (`/api/session/<id>/set_difficulty/`), перехода между этапами (`/api/session/<id>/advance_stage/`) и сброса сессии (`/api/session/<id>/reset/`).

### 3.7. Модуль редактора схем

Модуль редактора схем обеспечивает возможность создания и редактирования схем для физических задач. Схемы используются в этапах решения задач, где ученик должен нарисовать схему условия задачи или дополнить существующую схему.

**Функциональность модуля:**

Модуль предоставляет библиотеку элементов схем (точки, векторы, тела, оси координат и др.), которые учитель и ученик могут использовать для построения схем. Элементы организованы по категориям для удобной навигации. Учитель может создавать собственные элементы схем с настраиваемыми свойствами (цвет, размер, подпись и др.).

Редактор схем реализован на основе библиотеки Konva.js и предоставляет интерактивный холст для рисования. Ученик может добавлять элементы на схему, перемещать их, изменять свойства и удалять. Для эталонных решений схемы сохраняются в формате JSON, что позволяет точно воспроизводить их при отображении.

Модуль поддерживает два режима работы: режим редактирования (для создания схем) и режим просмотра (для отображения эталонных решений). В режиме просмотра элементы схемы не могут быть перемещены или изменены.

**API эндпоинты:**

Модуль предоставляет эндпоинты для получения категорий и элементов схем (`/api/schema/categories/`, `/api/schema/elements/`), сохранения схем ученика (`/api/schema/student/`), управления элементами и шаблонами схем через панель учителя.

### 3.8. Модуль панели учителя

Модуль панели учителя предоставляет интерфейс для управления всем контентом системы. Панель учителя реализована как отдельное React-приложение с расширенными возможностями редактирования.

**Функциональность модуля:**

Панель учителя включает несколько разделов: управление структурой курса (классы, разделы, темы), редактирование систем знаний, управление задачами, редактирование осмысления (зоны, вопросы, тексты с пропусками), управление методом решения и редактирование элементов схем.

В разделе редактирования систем знаний учитель может загружать изображения, настраивать параметры прохождения, создавать и редактировать зоны на изображениях, добавлять вопросы и тексты с пропусками. Интерфейс редактирования зон реализован с возможностью визуального выделения областей на изображении и сохранения их координат.

Раздел управления задачами позволяет создавать задачи, устанавливать правильные ответы, загружать эталонные решения и создавать пошаговые решения с использованием текста или схем. Редактор пошаговых решений поддерживает выбор типа шага (текст или схема) и наследование схемы от предыдущего шага.

**Особенности реализации:**

Панель учителя использует те же API эндпоинты, что и интерфейс ученика, но с дополнительными правами доступа. Все операции редактирования требуют аутентификации и проверки прав доступа (только пользователи с флагом staff могут использовать панель учителя).

### 3.9. Модуль интерфейса ученика

Модуль интерфейса ученика реализует все этапы обучения согласно методологии поэтапного формирования умственных действий. Интерфейс обеспечивает последовательное прохождение этапов: осмысление системы знаний, формулировка типовой задачи, решение задач, составление метода, пооперационный контроль и завершение обучения.

**Функциональность модуля:**

Интерфейс ученика автоматически определяет текущий этап обучения на основе сессии и отображает соответствующий компонент. На этапе осмысления ученик взаимодействует с изображением системы знаний, сопоставляя вопросы с зонами и заполняя пропуски в текстах. После успешного прохождения осмысления ученик переходит к формулировке типовой задачи.

На этапе решения задач ученик выбирает уровень сложности (легкое, непростое, трудное) и получает задачи в соответствии с адаптивным алгоритмом. После решения каждой задачи ученик видит эталонное решение и может продолжить или завершить этап.

Этап составления метода требует от ученика заполнить пропущенные шаги метода решения, выбрав правильные названия из предложенных вариантов. Этап пооперационного контроля позволяет ученику решить задачу по шагам, заполняя каждый шаг текстом или рисуя схему.

**Особенности реализации:**

Интерфейс ученика использует lazy loading для оптимизации загрузки компонентов, Context API для управления глобальным состоянием и мемоизацию для оптимизации производительности. Все взаимодействия с backend осуществляются через REST API с обработкой ошибок и индикацией загрузки.

### 3.10. Backend API модуль

Backend API модуль обеспечивает взаимодействие между frontend компонентами и базой данных. Модуль реализован на Django REST Framework и предоставляет RESTful API для всех операций системы.

**Функциональность модуля:**

Модуль включает ViewSets для всех основных сущностей системы: каталога, систем знаний, задач, сессий обучения, элементов схем и панели учителя. Каждый ViewSet реализует стандартные операции (list, create, retrieve, update, destroy) и кастомные действия через декоратор @action.

API использует сериализаторы для преобразования моделей Django в JSON и обратно, обеспечивая валидацию данных и оптимизацию запросов к базе данных через select_related и prefetch_related. Аутентификация реализована через сессии Django с поддержкой CSRF защиты.

Модуль включает обработку ошибок с возвратом понятных сообщений, валидацию входных данных и оптимизацию запросов для минимизации количества обращений к базе данных.

**Особенности реализации:**

API модуль разделен на эндпоинты для ученика и учителя, что обеспечивает разделение прав доступа. Эндпоинты учителя требуют дополнительной проверки прав доступа (флаг staff). Все операции с файлами (загрузка изображений) реализованы через FormData с поддержкой мультипарт-запросов.

## 4. ИНТЕГРАЦИЯ МОДУЛЕЙ

Все модули системы интегрированы через единый REST API, что обеспечивает слабую связанность компонентов и возможность независимой разработки и тестирования. Frontend компоненты взаимодействуют с backend через стандартные HTTP запросы с использованием fetch API.

Интеграция обеспечивается настройкой CORS для разрешения запросов между frontend и backend, использованием сессий Django для аутентификации, передачей CSRF токенов в заголовках запросов и синхронизацией состояния между клиентом и сервером.

Обработка ошибок реализована на всех уровнях: на уровне API возвращаются понятные сообщения об ошибках, на уровне frontend ошибки отображаются пользователю в удобном формате, а критические ошибки логируются для последующего анализа.
